# Рабочее место помощника арбитражного управляющего (`apps/procedure`)

Самый крупный раздел SiriCRM (веб-аналог программы «ПАУ»). Ведёт дело о банкротстве физлица (БФЛ, 127-ФЗ гл. X) от подачи заявления до завершения и списания долгов: стадии, процедуры, сроки-мероприятия, данные должника/супруга, документы, корреспонденция, публикации, интеграции с судом/ЕФРСБ/Коммерсантъ.

🛑 **Раздел делается строго по указаниям Дениса, поэтапно. Ничего не выдумывать — всё «оптимальное/нужное» согласовывать** (память `strict-direction-procedure`, `no-inventing-business-logic`). Сроки/состав мероприятий — ДАННЫЕ (правит АУ в UI), не хардкод.

**Статус: Этап 1 — на dev и на проде (выкачен 15.06.2026).** Дальнейшие этапы — см. роадмап внизу.

---

## Предметная область (как мыслит юрист/АУ)

Дело о банкротстве идёт по фазам:
1. **Общая фаза дела** (стадии: Подготовка/сбор документов → Подача заявления в суд → Принятие судом / первое заседание). Заканчивается **итогом первого заседания**.
2. Суд на первом заседании: **ввести реструктуризацию**, **ввести реализацию имущества**, **отказать**, или **мировое**.
3. **Процедура(ы)** — у дела бывает сразу реализация, либо сначала реструктуризация → потом реализация (т.е. процедур может быть несколько). У каждой свои стадии, даты (введение, публикации ЕФРСБ/Коммерсантъ), ФУ, исход.
4. При введении процедуры суд **назначает финансового управляющего (ФУ)**.
5. Обязательные **публикации** (ЕФРСБ + Коммерсантъ — две разные даты), реестр требований кредиторов (РТК), запросы в госорганы, собрания кредиторов, торги, отчёт ФУ.
6. **Завершение** — терминальный исход закрывает дело (списание долгов и т.п.).

Работа по отделам компании: Коммерческий (Власов) → Сбор документов (Нестеркина) → Юридический/АУ (Каныгина); общий контроль — Денис.

## Архитектура: двухуровневая модель

```
crm.Service (БФЛ)
   └─1:1─ BankruptcyCase (Дело)          ← общие стадии + итог 1-го заседания + общие даты
            ├── Procedure (Процедура 1)   ← вид, стадии, даты, ФУ, исход
            ├── Procedure (Процедура 2)   ← (если реструктуризация → реализация)
            └── ProcedureMilestone[]       ← мероприятия-сроки (привязаны к делу, опц. к процедуре)
```

Каталоги (редактируются в админке/UI, общие на инстанс):
- **`ProcedureStage`** — упорядоченный каталог стадий. `kind_scope`: `common` (стадии дела) / `restructuring` / `realization`. `is_terminal` — стадия «Завершено». Сид-коды: `prep`, `filing`, `accept` (common); `restr_start/restr_run/restr_done`; `real_start/real_auction/real_done`; `closed` (terminal).
- **`MilestoneTemplate`** — каталог мероприятий по стадиям. 🛑 Сроки тут как **данные**: `base_date_key` (якорь) + `offset_days` → дедлайн = базовая дата + N дней. `is_draft=True` — бейдж «срок не подтверждён АУ». Сейчас сид — DRAFT-заглушка (14 шаблонов), реальный перечень вносит Каныгина.

### `BankruptcyCase` (дело)
- `service` OneToOne (`related_name="bankruptcy_case"`), `status` (active/closed).
- `current_stage` (FK ProcedureStage) + `current_procedure` (FK Procedure) — текущее положение, подсвечивается в таймлайне.
- **Общие даты дела** (якоря общих стадий): `filing_date` (подача иска), `claim_accept_date` (приём иска), `first_hearing_date` (первое заседание).
- `first_hearing_outcome` (FIRST_HEARING_OUTCOMES).
- `fm_display` (property) — ФУ дела = ФУ **последней (актуальной) процедуры** (ФУ назначается судом при введении, поэтому живёт на процедуре, а в сводке показываем актуальный).
- `result_label` (property) — текстовый итог закрытого дела.

### `Procedure` (процедура)
- `case` FK, `kind` (restructuring/realization), `order`, `current_stage`.
- `financial_manager` (FK core.Employee, штатный) **или** `fm_name_external` (внешний АУ). `fm_display` собирает ФИО.
- **Даты процедуры** (якоря её мероприятий):
  - `intro_date` — «Дата принятия решения о введении процедуры» (= эффективное начало; в свёрнутом заголовке показывается `intro_date — end_date`).
  - `publication_efrsb_date` — «Публикация ЕФРСБ».
  - `publication_kommersant_date` — «Публикация КоммерсантЪ». 🛑 **ЕФРСБ и Коммерсантъ — две РАЗНЫЕ даты**, обе нужны (поле `start_date` и единое `publication_date` удалены — миграция 0007).
  - `next_hearing_date`, `term_months` (4–6), `end_date` («решение об окончании/завершении»).
- `outcome` (RESTRUCTURING_OUTCOMES / REALIZATION_OUTCOMES).

### `ProcedureMilestone` (мероприятие-срок)
- Принадлежит **делу** (`case`); `procedure` nullable (null = мероприятие общей фазы).
- **Снапшот** правила из шаблона (`title`/`base_date_key`/`offset_days`/`is_mandatory`) — правка каталога не переписывает живые мероприятия.
- `due_date` (вычисляется), `status` (pending/done/overdue/na/skipped), `is_manual`, `responsible`, `done_at/done_by`.
- Generic FK `artifact` (ct+uuid) — задел на привязку к будущему документу/корреспонденции/публикации.
- UniqueConstraint по (procedure, template) и (case, template при procedure=null) → идемпотентность инстанцирования.

### Исходы и автозакрытие дела
- Перечни: `FIRST_HEARING_OUTCOMES`, `RESTRUCTURING_OUTCOMES`, `REALIZATION_OUTCOMES`. `outcomes_for_kind(kind)` отдаёт нужный.
- `CLOSING_OUTCOMES` — терминальные коды (отказ, мировое, любой исход реализации, прекращение реструктуризации). `recompute_case_closed(case)` закрывает/переоткрывает дело по итогу 1-го заседания или исходу любой процедуры.

## Движок (`services.py`) — чистые функции над ORM

- `ensure_case(service)` — get_or_create, **ленивое создание на первом открытии карточки** (без бэкфилла услуг). На создании ставит первую `common`-стадию + инстанцирует её мероприятия.
- `instantiate_stage_milestones(case, stage, procedure=None)` — создаёт мероприятия по активным шаблонам стадии (идемпотентно), снапшотит поля, пересчитывает сроки.
- `recompute_due_dates(case)` — `due_date = base + offset_days` из снапшота. `_resolve_base_date` резолвит ключ: `case_*` от полей дела, `proc_*` от процедуры мероприятия.
- `add_procedure(case, kind, intro_date=…)` — создаёт процедуру, делает активной, переходит на её первую стадию, инстанцирует мероприятия, пишет событийку `procedure_added`.
- `enter_stage(case, stage, procedure=None)` — смена стадии (общей или процедуры) + инстанцирование + событийка `procedure_stage_changed`.
- `set_first_hearing_outcome` / `set_procedure_outcome` / `recompute_case_closed` / `close_case` / `reopen_case` — исходы + автозакрытие.
- `add_manual_milestone`, `set_milestone_status`.
- Лог/уведомления — только через `apps.crm.client_log` (`record_event`), EventType с `notifies=True` сам рассылает.

## Таймлайн стадий (в шапке карточки) — `_stages_bar.html`

Раздельный фрагмент, рендерится **в шапке справа от ФИО** (а не в теле вкладки — иначе перекрывал заголовок). Логика построения — `build_timeline_phases(case)` (отдельная функция во `views.py`, вызывается вьюхой `stages_bar`).

- **Несколько таймлайнов по фазам:** «До введения процедуры» (общие стадии) · по таймлайну на каждую процедуру (Реструктуризация/Реализация) · «Окончание» (терминал).
- **Горизонтальный**, названия чередуются сверху/снизу (`{% cycle 'up' 'down' %}`), дата стадии — на противоположной от названия стороне.
- 🛑 **Каждый узел шириной под своё название** (grid `1.35rem / 14px / 1.35rem` + слоты), линия — двумя половинками `::before/::after`, стыкуются у соседних точек → непрерывна и не зависит от длины подписей (длинные названия не налезают). Цвета: пройденные/текущая — синие, будущие — серые; текущая с кольцом.
- **Клик по стадии** → `set_stage` (переход), таргет `#procedure-tab-body`.
- 🛑 **Авто-обновление шапки:** шапка содержит `#proc-stages-bar` с `hx-trigger="load, procStagesChanged from:body"`. Любое действие в теле (через `_reload`) отдаёт `HX-Trigger: procStagesChanged` → блок перезагружается. Так таймлайн в шапке (вне `#procedure-tab-body`) синхронизируется со сменой стадий/дат/итога.
- **Привязка дат к стадиям** (`build_timeline_phases._stage_date`, поправимо): `prep`→обращение/анкета, `filing`→`filing_date`, `accept`→`first_hearing_date`; `*_start`→`intro_date`, `*_run`/`*_auction`→`publication_efrsb_date` (ЕФРСБ как основная), `*_done`→`end_date`; `closed`→дата окончания последней процедуры.

## Вкладка «Обзор / Сроки» (`_tab_overview.html`)

Двухколоночная сетка `2fr/3fr`: слева **«Дело»**, справа **«Данные должника»**. Ниже — **«Мероприятия и сроки»**. Перерисовывается целиком после любого действия.

### «Дело»
- Сводка: ФУ (read-only, из `case.fm_display`), **итог первого заседания** (select с авто-`hx-post`), даты услуги (Обращение/анкета RO, Подписание договора `date_dogovor`, Сбор документов RO, Подготовка иска RO) + даты судебного дела (Отправка иска `filing_date`, Принятие `claim_accept_date`/`first_hearing_date`).
- **Кастомный аккордеон процедур** (id `proc-acc-{case.id}`): на каждую процедуру — сворачиваемая карточка (header `data-proc-head`, шеврон, свёрнут показывает `intro_date — end_date` + бейдж исхода). Тело — форма процедуры (`update_procedure`): ФУ, и поля датами в раскладке
  ```
  Решение о введении процедуры     Следующее судебное заседание
  Публикация ЕФРСБ                 Публикация КоммерсантЪ
  Срок, мес.                       Решение об окончании/завершении
  Исход процедуры
  ```
  Кнопки Сохранить/Отмена/**Удалить** (dirty-tracking: активны только при правке). По умолчанию свёрнуто.
- **«+ Процедура»** (иконка plus, серая → зелёная): для **ПЕРВОЙ** процедуры вид определяется итогом 1-го заседания (Отказ/Мировое → кнопка неактивна; Реструктуризация → restructuring; Реализация → realization), для последующих — свободный выбор.

### «Данные должника» — табы «Должник» / «Супруг/а»
- Формы (`_person_form.html`): ФИО, ДР·Место рождения·ИНН·СНИЛС (одна строка), паспорт в рамке `<fieldset>`, сумма долгов (только Должник, = `Service.total_debt`). Сохранение — `update_person` (правит карточку `Client`). Dirty-tracking кнопок.
- **Телефоны** — полный CRUD, scoped по `who` (`_phones_block.html`, через `apps.crm.phone_utils`).
- **Адреса** — полный CRUD с DaData-typeahead, scoped по `who` (`_address_block.html`/`_address_form.html`, скопировано из карточки клиента; id с префиксом `{{who}}` во избежание коллизий).
- **Супруг/а** (`Client.spouse`, self-FK, взаимная привязка + `is_married`):
  - Поиск (`spouse_search`, typeahead) — в результатах **только полное ФИО** (телефон не показываем; искать по телефону можно).
  - 🛑 **Выбор в два шага** (нельзя писать в БД сразу — можно ошибиться): клик по результату → `spouse_pick` рендерит превью «Выбран: ФИО» с кнопками **«Сохранить»** (→ `spouse_link`, пишет в БД) и **«Выбрать другого»** (очищает). Кнопка-иконка ＋ «добавить нового» → `spouse_create` (пустая запись `Client`, заполняется в полной форме).
  - 🛑 **После сохранения/правки супруга активным остаётся таб «Супруг/а»** (контекст `active_person_tab`, прокидывается через `_reload(..., person_tab="spouse")`; radio-табы `checked` по нему). Иначе перерисовка сбрасывала на «Должник».

### «Мероприятия и сроки»
- Список с подсветкой просрочки (`bg-error/10`), бейдж «просрочено: N», группировка Общие/по процедуре. Кнопки статуса (Выполнено/Не применимо/…), «Добавить мероприятие».

## Даты услуги (откуда берутся)

Часть дат — на уровне услуги/дела (1–7), часть — процедуры (8–11):
1. **Обращение/анкета** — дата анкеты услуги (`QuestionnaireResponse`), иначе `Client.created_at`.
2. Подписание договора — `Service.date_dogovor`.
3. **Передача в отдел сбора документов** — `Service.docs_dept_date` (crm 0087). Заполняется автоматически в `service_transfer.transfer_service`, когда услуга переходит в отдел с флагом **`Department.is_docs_collection`** (core 0024; в проде — «Отдел сбора документов БФЛ»).
4. Передача на подготовку иска — событие `claim_prep_assigned` (событийка).
5–7. `filing_date`, `claim_accept_date`, `first_hearing_date` — на деле.
8–11. `intro_date`, `publication_efrsb_date`, `publication_kommersant_date`, `next_hearing_date`, `end_date` — на процедуре.

Доп. поля в crm.Service: `total_debt` (0086), `docs_dept_date` (0087).

## Гоччи (не наступать повторно)

- 🛑 **`@never_cache`** на GET-партиалах — иначе «ничего не меняется» (браузерный кэш HTMX-фрагмента).
- 🛑 **Шаблоны кэшируются в prod** (`cached.Loader`) — после правки шаблона рестартить `web`; пользователю — `Ctrl+Shift+R`.
- 🛑 **Таймлайн — в шапке (вне `#procedure-tab-body`)** → обновляется только через `HX-Trigger: procStagesChanged`, который отдаёт `_reload`. Не забывать при добавлении новых действий, меняющих стадии/даты.
- 🛑 **Узлы таймлайна — ширина под название + линия половинками**; не возвращать фиксированную ширину/сегменты (длинные подписи налезают).
- 🛑 **daisyUI timeline/collapse классы вырезаны** из `static/css/tailwind.css` (PURGE) — таймлайн и аккордеон сделаны на своём CSS/JS. Новые tailwind-классы проверять `grep` в tailwind.css или ставить inline-style.
- 🛑 **Инлайн-`<script>` в партиалах — в IIFE** (top-level const падает при повторном свопе).
- 🛑 **Супруг: запись в БД только по явной кнопке**, поиск/выбор ничего не пишут. После операций с супругом — таб «Супруг/а» активен (`active_person_tab`).
- 🛑 **Адреса/телефоны scoped по `who`** (id с префиксом) — иначе коллизии между формами должника и супруга.
- 🛑 **ФУ — на процедуре** (назначается судом при введении), в сводке дела — актуальный (последней процедуры).
- 🛑 **Сроки мероприятий — данные** (`MilestoneTemplate`), не хардкод в Python.
