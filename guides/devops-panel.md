# DevOps-панель — инструкция пользователя

Сервисная панель для управления SiriCRM: бэкапы базы, перенос данных между окружениями, деплой новых версий, мониторинг S3.

## Кому доступна

- Открывается только на dev-сервере: **https://crmsiri.ru/devops/**
- Только пользователю с правом **суперюзера** (в админке Django — галочка «Является суперпользователем»). Всех остальных Django редиректит на логин и не пускает.
- На prod-сервере (siricrm.ru) панель тоже технически развёрнута, но из браузера её **открывать не надо** — она здесь работает только как HTTP-агент, принимающий команды с dev.

## Что видно на главной

Дашборд разделён на 5 секций:

1. **Состояние серверов** — карточки по каждому окружению (dev/prod): текущий коммит, отставание/опережение по git, число живых контейнеров, ожидающие миграции, занятость диска.
2. **Базы данных (PostgreSQL)** — копирование данных между prod и dev, бэкапы, список бэкапов в S3.
3. **Деплой** — выкатить новые изменения (отдельные кнопки на dev и на prod), пересборка образов, откат на предыдущую версию.
4. **S3-хранилище** — статистика по бакетам (медиа, бэкапы) с разбивкой по префиксам.
5. **История действий** — последние 25 запусков с их статусом, кто запустил, когда.

История обновляется автоматически раз в 10 секунд.

## Как читать состояние сервера

В карточке окружения:

- **git: `<ветка>` @ `<коммит>`** — на каком коммите сейчас сервер.
  - `· незакоммичено!` (жёлтое) — на сервере есть несохранённые правки в файлах. Деплой и откат откажут — сначала разберитесь с этими правками руками.
  - `· отстаёт на N — есть что деплоить` (жёлтое) — на удалённом репозитории (GitHub) есть N новых коммитов. Жмите «Задеплоить».
  - `· в синхроне` (зелёное) — всё актуально.
- **контейнеры: 10/10 up** — сколько запущено / всего объявлено в docker compose. Красное число — что-то упало.
- **миграции ждут: N** — есть ли неприменённые миграции Django. Деплой их применит автоматически.
- **диск: 65% (12G свободно)** — заполненность диска. Жёлтое от 70%, красное от 85%.

Если карточка пустая — нажмите **↻ Проверить состояние**. Кнопка **↻ Обновить** там же — пересобрать данные.

## Базы данных

### Копирование данных

Кнопка **«prod → dev (затянуть боевые данные)»** — стянуть свежий дамп БД с продакшна на dev. Удобно когда нужно отладить багу на реальных данных. Действия:

1. Подтвердить кодовое слово **PULL**.
2. На стороне dev автоматически сделается защитный бэкап текущей dev-БД (всё равно, кому она была нужна — на всякий случай).
3. Затем берётся свежий дамп с prod, выливается на dev. **Все текущие dev-данные удаляются.**
4. По итогам — структурированный лог: размер дампа, время заливки, ссылка на защитный бэкап.

Кнопка **«dev → prod (залить dev в боевую)»** — обратное направление. Это потенциально разрушительная операция, поэтому защит больше:

1. Подтвердить кодовое слово **PUSH-PROD**.
2. Поставить дополнительную галочку «Я понимаю последствия».
3. На стороне prod автоматически делается защитный бэкап текущей боевой БД. Если защитный бэкап не получился — операция прерывается, prod не трогается.
4. Затем выливается дамп с dev. **Все текущие данные на проде удаляются.**

Используйте `push_db` только если уверены: проще выкатить миграции через деплой, чем затирать боевую БД дев-данными.

### Бэкапы

- **📦 Бэкап БД: dev / prod** — однократный бэкап текущей БД. Дамп жмётся gzip, заливается в S3 в бакет с бэкапами (см. `AWS_BACKUP_*` в env), отдельно остаётся локальная копия в контейнере `backup`. После завершения в результате будет пре-подписанная ссылка для скачивания.
- **📋 Бэкапы: dev / prod** — посмотреть список последних бэкапов в S3 с размерами и датами.

Помимо ручных бэкапов на каждом сервере крутится контейнер `backup`, делающий бэкап ежедневно (внутреннее расписание сервиса).

## Деплой

### Обычный деплой

**🚀 Задеплоить на dev / Задеплоить на prod** — стандартный пайплайн:

1. `git fetch` + `git pull --ff-only` на нужной ветке (по умолчанию `feat/production-ready`). Если pull не «fast-forward» (на сервере есть локальные коммиты, которых нет в origin) — операция откажется.
2. `python manage.py migrate --noinput` — применить миграции.
3. Перезапуск контейнеров `web`, `celery`, `celery-beat`. **Короткий даунтайм** — обычно 5–10 секунд, на rebuild — до 2–5 минут.
4. Healthcheck `http://web:8000/health/` — должен вернуть 200 и `{"status":"ok","db":true,"redis":true}`. Если не вернёт за 5 попыток — деплой помечается failed.

Подтверждение: кодовое слово **DEPLOY**. В поле «branch» можно указать другую ветку, если деплоите экспериментальную.

> **Важно:** деплой **не перезапускает контейнер `devops-runner`** на цели. Если в коде поменялись хендлеры/таски (`apps/devops/handlers/*` или `apps/devops/tasks.py`), runner останется на старом коде и новые типы действий будут падать с «Неизвестный action_type». В этом случае на нужном сервере вручную: `docker compose -f <compose> --env-file <env> restart devops-runner`. См. пути в [CLAUDE.md](../CLAUDE.md).

### Rebuild (пересборка образов)

**🔨 Rebuild: dev / prod** — нужен только если поменялся `Dockerfile`, `requirements.txt`, или сама структура образа. Это `git pull` + `docker compose build` + `up -d --force-recreate`. Дольше обычного деплоя, даунтайм 2–5 мин. Контейнеры `db`, `redis`, `devops-runner` не пересоздаются.

### Откат версии

1. Нажмите **📜 Версии dev / prod** — откроется список последних 30 коммитов в репозитории.
2. Возле каждого коммита (кроме текущего HEAD) — кнопка **↶ Откатиться сюда**.
3. Подтвердите — выполнится:
   - `git reset --hard <коммит>` (на сервере; **локальные изменения там пропадут**, поэтому при «незакоммичено!» в статусе откат откажется);
   - Попытка реверса миграций, которые появились между целевым коммитом и текущим HEAD. Если хотя бы одна миграция irreversible — **откат прерывается до перезапуска контейнеров**, БД остаётся в текущем состоянии, и в логе будет понятно, какая миграция мешает. Дальше — руками: либо реверсните руками всё что можно и удалите проблемную миграцию, либо восстановите БД из бэкапа.
   - Перезапуск `web`/`celery`/`celery-beat` + healthcheck.

## S3-хранилище

**📊 Бакеты dev / prod** — посчитать сколько занимают бакеты этого окружения. Разбивка по префиксам (`media/`, `documents/`, `backups/`, …). Для prod запрос уйдёт на агента siricrm.ru — увидите только если на проде включен агент и токен в `DEVOPS_AGENT_TOKEN_PROD` валидный.

Backup самих бакетов в S3 — пока не реализован, в roadmap.

## Что делать, если действие зависло

### «Долго работает»

Все действия выполняются асинхронно. Карточка действия (`/devops/actions/<id>/`) опрашивает статус раз в 2 секунды через HTMX, плюс параллельно крутится фоновый sync в Celery (`devops.sync_action`) — даже если вы закрыли вкладку, статус подтянется не позже чем через 3 секунды после завершения на стороне агента.

Если действие идёт больше **60 минут** — фоновый sync принудительно гасит его с пометкой «зависло на стороне агента». Это страховка от навсегда-running записей.

### «Не удалось запустить»

В заголовке карточки увидите красную плашку. Самые частые причины:

- **«Неизвестный action_type»** — на целевом сервере `devops-runner` работает на старом коде. Решение: см. примечание про runner в разделе про деплой.
- **«Агент вернул 401»** — невалидный или отсутствующий `DEVOPS_AGENT_TOKEN_PROD` в `.env.dev`. Сравните с тем что лежит на проде в `DEVOPS_AGENT_TOKEN`.
- **«Connection refused / timeout»** — целевой сервер недоступен или у него лежит nginx. Проверьте `curl -I https://siricrm.ru/health/` напрямую.
- **«Failed to fast-forward»** — на сервере есть локальные коммиты, которых нет в origin. Кто-то правил код на самом сервере; разбирайтесь вручную (`git status`, `git log`) на этом сервере.

Полный текст ошибки и стектрейс — в логе действия (вкладка «лог» в карточке).

## Опасные действия — устройство защит

«Опасных» действий пять: `pull_db`, `push_db`, `deploy`, `rebuild`, `rollback`. Каждое требует подтверждения в модалке:

- **Кодовое слово** — нужно ввести ровно символ-в-символ. Кнопка submit активируется только при совпадении (`PULL`, `PUSH-PROD`, `DEPLOY`, `REBUILD`, `ROLLBACK`).
- **Дополнительная галочка** — только у `push_db`. Без неё submit не активируется даже при правильном слове.
- **Автоматический бэкап** — `push_db` всегда сначала бэкапит цель. Если бэкап не удался — операция отменяется (БД цели не трогаем).
- **Defence-in-depth для деплоя** — `--ff-only`, поэтому случайно не «переписать» историю на сервере; healthcheck в конце — если новый код упал, увидите failed.

## История действий и логи

В нижней секции дашборда — последние 25 запусков. Колонки: время, окружение, тип действия, статус, кто запустил.

- Зелёный значок — done, красный — failed, серый крутящийся — running.
- Клик по строке → детальная карточка с полным выводом stdout, структурированными результатами (где есть — например, для `status`, `s3_stats`, `git_log`) и кнопкой «↺ Повторить» для безопасных readonly-действий.

Глубже истории нет — нужно посмотреть старые действия, открывайте через админку: `/admin/devops/devopsaction/`.
