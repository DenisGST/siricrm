# Журнал объединений клиентов (дубли)

Лог проведённых мерджей клиентов в Siri. Пополняется при каждом ручном объединении. Помогает увидеть повторяющиеся паттерны (например, как часто всплывает «WhatsApp-автолид + менеджерская карточка») и где стоит автоматизировать.

## Паттерны (краткая классификация)

| Тип | Описание | Лечение | Где описано подробнее |
|---|---|---|---|
| **WA-autolead+manual** | Менеджер занёс лида с грязным форматом телефона (`+7 (914) 5555918`), клиент ответил в WA → нормализация создала второго `first_name="WhatsApp"`. У второго есть ClientPhone, больше messages | canonical = менеджерская карточка с полным ФИО, переносим WA-чат, services сливаем (CanonicalService = со статусом) | [[memory:wa-leads-merge-pattern]] |
| **TG-lead+main** | Клиент написал в TG (Telethon userbot создал лид c `telegram_id`+`username`), но он уже есть в Siri (полная карточка, WA-чат) | canonical = полная карточка, src = TG-лид. Перенести `tg_id`+`username`+TG-messages. UNIQUE на `telegram_id` — обнулить у src перед save | Аналогично, см. кейсы Танатаркин, Соловьёва, Никитина |
| **MAX-lead+main** | Клиент написал в MAX → новый лид с `max_chat_id`+username, при существующей карточке | canonical = полная карточка, перенести `max_chat_id` (UNIQUE — обнулить у src сначала) + MAX-messages | См. Токманцева Галина, Миргалеева Ольга/Оля |
| **Triple** | TG + MAX + основная WA-карточка одного клиента (3 канала независимо создали лидов) | canonical = основная (с услугой/charges/payments), переносим tg_id+username из TG, max_chat_id из MAX, messages из обоих | Никитина Наталья Олеговна |
| **Bubble re-fetch / Bubble дубль Man** | В Bubble на одного клиента **2 Man-записи** (менеджер случайно дублировал). Обычно одна импортирована, вторая зависает в pending → её PBFL/Money/Correspondence ошибаются | apply Bubble Man с `overrides.merge_into_client_id=<siri>` + если phone=0000 то `phone_override`. Затем apply дубль-PBFL, потом слить две Service у клиента (см. [[bubble_integration_claude.md]] раздел 9) | Кузина Наталья, Трифонова Арина |
| **Family-on-same-phone** | На одном номере действительно живут разные люди (супруги, родители) | **НЕ мерджить!** Подсветить пользователю, оставить как есть | Беликовы, Разуваевы, Сусловы, Палеевы (4 июня) |
| **CAPS-typo** | В Bubble менеджер создал запись КАПСОМ с опечаткой → через 14 мин пересоздал нормально. Скипнуть первую | `BubbleRecord.status='skipped', approved=False, error='Дубль с опечаткой, пересоздан как ...'` | Крокос Диана ([4 июня]) |
| **Опечатка ФИО** | В Siri опечатка в фамилии/имени (например «Владимслав»), в Bubble правильно | apply Bubble Man с `overrides.merge_into_client_id=<siri>` + **`force_rename_existing=True`** → перезапишет ФИО из raw | Трифонов Владислав (5 июня) |

## Стандартный поток мерджа (Siri+Siri)

См. [[memory:wa-leads-merge-pattern]] и [[memory:charge-payment-client-fk]] (защитный перенос `Payment.client`/`Charge.client`!). Кратко:

1. Освободить UNIQUE-поля у src (`telegram_id`/`max_chat_id`/`whatsapp_phone`/`username` → None у src перед save на tgt)
2. ClientEmployee: дубли удалить, уникальные перенести
3. ClientPhone: те же phone+purpose у tgt — удалить у src, остальные переключить
4. Services: если у обоих по 1 → выбрать canonical (со статусом / с bubble_id), добавить emps другого, перенести Charge/Payment, удалить второй
5. FK перенос: Message, ClientLogEntry, EmployeeLog, Payment.client, Charge.client (защитно)
6. Folders: переключить `client`, потом рекурсивный `merge_subtree` корней (по `name` + `parent`)
7. `src.delete()`

## Массовые операции (не мерджи клиентов)

Сюда — операции переноса/синхронизации сразу по многим услугам или клиентам (не индивидуальные мерджи). Например: миграция отдела с Bubble на Siri, массовый rebalance канбана, перенос файлов под другую папочную структуру.

#### Технический фикс на dev (не катился на prod)

**`apps/crm/client_merge.py:_move_files`** переписан — был баг: `null value in column "folder_id"` при мердже через UI, если у survivor 0 папок (`surv_root = None` → `cf.folder = None` → IntegrityError). Также старая логика плоско сваливала все файлы в root, теряя структуру. Новый `_move_files`:
1. `get_or_create_root(survivor)` — гарантируем корневую папку
2. `ClientFolder.objects.filter(client=source).update(client=survivor)` — массовое переключение всех папок source, **структура вложенности сохраняется**
3. После переноса — рекурсивный `_merge_folder_subtree` для одноимённых корней (та же логика что в моих скриптах)
4. Имя итогового корня нормализуется по ФИО survivor

Готов к выкатке на prod через DevOps-агента — пользователь подтвердит.

## Что использовать как «не сливать»

- **Family-on-same-phone** — разные ФИО на одном номере, разные дни рождения. Например 4 июня: Беликовы (Оксана+Екатерина), Разуваевы (Данила+Светлана), Сусловы (Андрей+Оксана), Палеевы (Любовь+Павел). При диагностике (`for c in phone_to_clients[phone]: if cnt>=2 ...`) — оператор должен подтвердить что это супруги/родственники, иначе они **выглядят как дубль по номеру**.
- **Однофамильцы из разных мест** — Бондаренко Екатерина (Григоренко, Волгоград) ≠ другая Бондаренко уже в Siri. При apply Bubble Man с пустым phone applier ФИО-совпадение НЕ ловит, создаст отдельную → ОК.
- **Тестовые** — `+79999999999` или fake `+790000000XX` — это маркер «тест/агентский», по ним не считать дублями.
