# Дедупликация клиентов + нормализация телефонов

Защита от дублей клиентов и нормализация номеров. Кратко в `CLAUDE.md`.

## Правила (согласовано с пользователем)
- **Телефон — сильный ключ: один номер = один клиент.** При ручном создании/редактировании дубль по телефону **блокируется**. ФИО — слабый сигнал (однофамильцы): только **предупреждение**, создать можно (двойники с разными телефонами допустимы).

## Точки создания клиента и дедуп
| Где | Дедуп |
|---|---|
| Кнопка «Создать клиента» (`client_create`/`ClientForm`) | ✅ блок по телефону + ФИО-хинт |
| Редактирование (`client_edit` + блок телефонов `client_phone_add`) | ✅ блок по телефону + ФИО-хинт |
| WhatsApp webhook (`_get_or_create_wa_client`) | ✅ по телефону |
| Telegram leads-bot (`create_lead_from_parsed`) | ✅ по телефону |
| MAX webhook (`get_or_create(max_chat_id)`) | ⚠️ только по chat_id (MAX телефон не даёт) — **на будущее** |
| Telegram userbot (`get_or_create(telegram_id)`) | ⚠️ по telegram_id, не по телефону — **на будущее (Этап 3)** |
| Анкета (`questionnaire`) | ⚠️ без дедупа — **на будущее (Этап 2)** |

Услуги: `route_new_lead` → `Service.get_or_create(client, name)` — один БФЛ на клиента, дублей не плодит.

## Реализация
- **Источник правды дедупа — `ClientPhone` (нормализованные цифры `79991234567`).** Поиск — `find_client_by_phone` (`phone_utils.py`). Backfill `crm.0065` дал ClientPhone историческим; при ручном сохранении заводится `ClientPhone(primary)`.
- **`ClientForm.clean_phone`** — блок дубля (ClientPhone + fallback по legacy `Client.phone`, исключая себя при edit); хранит номер в формате `+7 (XXX) XXX-XX-XX` (`format_phone`). При конфликте ставит `form.phone_conflict` → баннер со ссылкой «открыть».
- **`client_phone_add`** — блок номера, занятого ЛЮБЫМ другим клиентом (не только тот же purpose), с сообщением.
- **Лайв-хинты `client_dedup_check`** (`/clients/dedup-check/`): по мере ввода (от 3 символов) oob-свапы в `#dedup-fio` / `#dedup-phone` (всплывают у полей). ФИО — список похожих (информационно). Телефон — частичные совпадения + при полном конфликте красный баннер и **disable кнопки «Создать»** (oob по `#client-create-submit`). `exclude=<client.id>` для формы редактирования (не показывать себя, свой номер не конфликт). Контейнер рендерится по НАЛИЧИЮ параметра в запросе (ввод телефона не затирает ФИО-хинт). Кнопка «Открыть карточку» удаляет create-модалку в `after-request` (в `onclick` отменялся бы ещё не ушедший hx-get).
- **Нормализация номера:** `format_phone(raw)` → `+7 (XXX) XXX-XX-XX` (8→7, скобки/пробелы/`@c.us` срезаются через `normalize_phone`). JS `window.formatPhoneField(el)` (dashboard.html) форматирует поле на `onblur`. `sync_client_phone_cache` тоже кладёт `Client.phone` в красивом виде.

## Карточка клиента: ОДНА кнопка «Сохранить» на все вкладки
`client_edit_modal.html` — одна пустая `<form id="client-edit-form">` (только csrf), а все поля вкладок (Основное/Документы/Заметки) связаны с ней HTML-атрибутом `form="client-edit-form"` (htmx сериализует через `form.elements`). Блоки **телефонов** и **адресов** имеют СВОИ `<form>` (отдельный CRUD) и в основную форму не входят — поэтому привязка атрибутом, а не вложением (вложенные `<form>` недопустимы).

## 🛑 Гочча: `<input type="date">` требует ISO (иначе затирает даты)
`<input type="date">` принимает значение ТОЛЬКО как `YYYY-MM-DD`. Если рендерить `value="{{ form.birth_date.value }}"`, при русской локали Django выдаёт «18 января 1977 г.» → поле показывает **пусто** → сохранение (особенно единой кнопкой, шлющей все поля) **затирает дату в БД**. **Всегда рендерить даты как `value="{{ form.X.value|date:'Y-m-d' }}"`** (поля `birth_date`, `passport_issued_date` в create/edit модалках). Был инцидент 04.06.2026: даты рождения/выдачи паспорта пропадали после сохранения.
